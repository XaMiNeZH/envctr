# Versioning Strategy

envctr uses semantic versioning with this format:

```
MAJOR.MINOR.PATCH
```

- `MAJOR` increases when an option or behavior changes in a breaking way.
- `MINOR` increases when a new compatible feature is added.
- `PATCH` increases when bugs are fixed without changing behavior.

## Current Version

The current version is stored in the `VERSION` file at the repository root.
Running `envctr --version` reads that file so the command line and repository
always match.

## Release Workflow

1. Update `VERSION`
2. Add a dated entry to `CHANGELOG.md`
3. Commit with a clear message referencing the change
4. Create a Git tag `vX.Y.Z`
5. Push commits and tags

## Branch Strategy

- `main` — stable, submission-ready code only
- `dev` — active development branch
- `feature/<name>` — individual feature branches, merged via PR into `dev`
- `dev` is merged into `main` when a milestone is complete

## Pull Request Rules

- Every feature must be a PR from a feature branch into `dev`
- PR description must reference the requirement it implements
- At least one teammate must review before merge
- No direct commits to `main`

## Why Versioning Matters

Every lockfile generated by envctr records the `envctr_version` that created
it. This allows teammates to detect when a lockfile was generated by an older
version of the tool and may need to be regenerated.
